# Contribuir a ACDP (Protocolo de Coordinación de Agentes)

¡Primero que nada, gracias por considerar contribuir a ACDP! 

Construir un protocolo robusto requiere disciplina técnica y pensamiento arquitectónico. Aquí priorizamos **los conceptos sobre el código** y **las bases sólidas sobre la inmediatez**.

## La Regla de Oro: Issue-First (El Problema Primero) 🚧

**No aceptamos Pull Requests sorpresivos.**

Si vas a proponer un cambio significativo, una modificación al protocolo o una mejora estructural, seguimos de forma estricta una metodología **Issue-First / Request for Comments (RFC)**.

1. **Discute el Problema**: Abre un Issue para plantear el problema que quieres resolver. Programar sin contexto o entendimiento fundamental es la solución equivocada.
2. **Propón la Solución (RFC)**: Describe tu enfoque arquitectónico en el Issue. Explica *por qué* es la solución adecuada y expón sus trade-offs.
3. **Consenso**: Espera a que un mantenedor del proyecto valide y apruebe el enfoque técnico.
4. **Ejecución**: Solo tras recibir la aprobación de tu enfoque, crea el código y envía el Pull Request mencionando el Issue aprobado.

### ¿Por qué trabajamos así?
Porque corregir un enfoque arquitectónico erróneo durante la revisión de un PR es frustrante y una pérdida de tiempo para ambas partes. Los cimientos sólidos se diseñan *antes* de la construcción. No hay atajos para el buen diseño de software.

## Proceso para Pull Requests

1. Asegúrate de que tu PR esté referenciado a un Issue previamente aprobado y discutido.
2. Usa **Conventional Commits** exclusivamente para tus mensajes de commit. Está prohibido el uso de firmas automáticas o menciones a IAs en los commits.
3. Actualiza cualquier esquema relevante (ej. `messages.schema.json`) y la documentación necesaria antes de enviar el PR.
4. Espera pacientemente la revisión. Evaluaremos tu implementación contra la arquitectura y enfoque técnico previamente acordados en el Issue.

Gracias por aportar a un estándar más sólido.
